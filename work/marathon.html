Marathon

Marathon launched with a UI that players immediately described as the game's biggest problem. Five or six menu levels to reach basic items. Loadout and Vault living in completely separate screens with no way to cross-reference them. A visual design that looked cool in screenshots but was nearly unreadable during play.

I ran a full UX audit — mapped pain points with severity ratings, pulled real player feedback from Steam and press, ran task flow analysis across four user journeys, then redesigned the entire interface from the ground up.

Research

Marathon's menu navigation was the game's most-criticised failure point. Players felt lost between disconnected screens, overwhelmed by visual clutter, and unable to complete basic tasks without passing through unnecessary layers. The UI was acting as an obstacle between players and the run.

P-01Critical
Navigation Depth
5 to 6 menu levels to reach common items. Vault, Codex, and Contracts are fully siloed with no cross-access.
P-02Critical
Disconnected Flows
Loadout and Vault are separate with no link. Players prep a run without being able to see available gear.
P-03Critical
Visual Overload
Every screen surfaces too much simultaneously. Players can't tell what's actionable from what's decorative.
P-04High
Tooltip Hunting
No comparison view. Reading stats requires hovering each item individually, forcing players to memorize between hovers.
P-05Medium
No Shortcuts
No way to jump directly to key screens. Every session means re-navigating the same deep paths from scratch.
P-06Medium
Controller Bias
UI appears designed controller-first. PC players get an awkward experience where mouse patterns feel unnatural.

Main Menu

The main menu became the prep room instead of a list of doors. I kept the runner in the center because that is the part of Marathon that already had a strong identity, then moved the actual decisions to the edges: run setup on the left, pressure cards and map state on the right, party and system actions in the bottom rail.

The goal was not to make a clean menu for the sake of it. It was to stop players from opening five different screens just to answer simple questions before searching: what am I doing, who is with me, what map is active, what contracts are waiting, and am I geared enough to go?

Before After
01 — Before / After Slider

Prepare Hub

What it is. The redesigned landing screen before a run. Prepare, factions, codex, map, contracts, party state, currency, and search are all in the same view.

Why it looks this way. The runner and the blue hangar stay as the visual anchor, while the useful pieces move into clear zones: navigation on the left, live pressure on the right, actions along the bottom. The green search button stays loud because that is the commit moment.

Issue it solves. The old home screen made players leave the lobby to answer basic prep questions. This version lets them understand the run, fix blockers, and search without bouncing through separate menus.

02 — Main Menu Detail

Map Info Hover

What it is. A quick information card for the selected map, showing threat, location, and a short description from the lobby.

Why it looks this way. It opens beside the map tile instead of taking over the screen, so the player understands it as extra context, not a new page.

Issue it solves. Players can check what they are about to queue into without leaving the prepare flow or guessing from a map name alone.

03 — Main Menu Detail

Loadout Preview

What it is. A compact loadout card that shows the active shell, weapons, core gear, and rough value directly from the hub.

Why it looks this way. It uses the same card language as the full inventory, just tightened into a small overlay. The player sees enough to confirm the build without opening Loadout.

Issue it solves. It removes the small but constant back-and-forth of checking gear before every search.

04 — Main Menu Detail

Game Mode Card

What it is. A mode card for ranked or timed activity states, paired with the active map tile.

Why it looks this way. The timer and active state sit on top because they change the decision. The image stays below so it still feels tied to the destination.

Issue it solves. Players know what queue is active and how urgent it is before they hit search.

05 — Social State

Party Stack

What it is. A visible party stack that shows teammates and their selection pressure cards while staying on the main screen.

Why it looks this way. The cards stay bright and vertical on the right because they are run blockers, not background info.

Issue it solves. The player can see who is ready, who is missing requirements, and what is stopping the party from searching.

06 — Main Menu Detail

Contract Peek

What it is. An expanded contract card that shows objectives and progress without opening the full contracts page.

Why it looks this way. It expands near the pressure stack, so the player can connect the warning to the exact task causing it.

Issue it solves. Players stop treating contract cards as vague alerts. They can read the blocker and act on it from the same place.

07 — Map Control

Map Select List

What it is. A small map switcher that opens from the active map tile and lists the available destinations.

Why it looks this way. The map list stays inside the tile footprint, so it feels like a dropdown with weight instead of a separate screen.

Issue it solves. Map swapping becomes part of prep, not another detour before matchmaking.

08 — Social Layer

Friend Profile

What it is. A focused profile panel with banner art, name, account IDs, join/invite actions, and safety options.

Why it looks this way. The background blurs instead of disappearing, and the profile uses a split image/info layout so it still feels tied to the lobby.

Issue it solves. Checking or inviting someone no longer feels like opening a separate social app inside the game.

09 — Social Layer

Friends List Drawer

What it is. A right-side drawer for crew, Marathon connections, Bungie connections, and offline friends.

Why it looks this way. It slides in over the lobby and keeps status groups stacked vertically, which makes scanning faster than a full-page friends screen.

Issue it solves. Party management becomes a quick side action, not something that breaks the flow of preparing for a run.

Inventory & Loadout

The original Loadout and Vault treated the two most connected actions as separate chores. You built a loadout in one place, checked your stored gear somewhere else, then tried to remember which weapon or mod was worth swapping.

I rebuilt it as one working table. Equipped gear sits on the left, the runner stays in the middle, and the vault lives on the right. The screen is heavier than the hub on purpose, because this is where players actually compare, tune, and make tradeoffs before a run.

Before After
01 — Before / After Slider

Unified Loadout + Vault

What it is. A single working space for the equipped loadout, backpack, shell slots, character, and vault.

Why it looks this way. I split the screen by task: current build on the left, character and slot targets in the middle, stored gear on the right. It is denser than the hub because this is the screen where players actually make tradeoffs.

Issue it solves. The original separated Loadout and Vault, so players had to remember items between screens. This puts ownership, equipment, and comparison in one place.

02 — Inventory Screen

Equipped Gear + Vault

What it is. The main inventory state with equipped weapons and backpack on the left, the runner in the center, and the vault grid on the right.

Why it looks this way. The left side answers “what am I carrying,” the right side answers “what do I own,” and the center keeps shell changes connected to the body.

Issue it solves. Players can prep a build without opening separate pages just to understand what is equipped versus stored.

03 — Inventory State

Item Inspect

What it is. A hover panel for a selected weapon, with stats, attached parts, value, and available actions.

Why it looks this way. The card opens beside the item grid instead of replacing the screen. The player keeps the vault context while reading the details.

Issue it solves. It cuts down tooltip hunting and lets players inspect items without losing track of where the item came from.

04 — Inventory State

Item Compare

What it is. A side-by-side comparison state for two weapons, with matching stat rows and attachment previews.

Why it looks this way. I gave the comparison its own dark center panel so both items get equal weight while the inventory stays visible behind it.

Issue it solves. The player does not have to memorize numbers from one hover to the next. The decision can happen on-screen.

Shell System

Customization needed to feel like part of preparation, not a separate cosmetics drawer. Shell choice changes stats, traits, abilities, and the way a player reads their role, so it deserved the same level of clarity as weapons.

The shell screens keep the visual attitude of Marathon, but give the player a cleaner way to understand what a shell does before committing to it.

01 — Shell System

Shell Select

What it is. A shell browser with large character art, a left-side roster, rarity markers, and a right panel for abilities and traits.

Why it looks this way. The shell gets the stage because it is both gameplay and identity. The ability panel stays separate so players can read the kit without fighting the artwork.

Issue it solves. Shell choice becomes understandable before equipping. Players can read role, personality, and function in one place.

02 — Shell System

Shell Customization

What it is. A detailed shell view for skin, source, style, and flavor text.

Why it looks this way. It is built more like a poster than a list: big preview first, readable lore panel second, equip/source actions kept close to the name.

Issue it solves. Cosmetics stop being tiny thumbnails. Players can judge the full look and understand where it came from without opening another menu.
